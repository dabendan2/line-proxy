# 全記憶重構協議 (Full Context Memory Reconstruction)

在移除了手動忽略邊界 (`--last-ignored-msg`) 機制後，代理人必須採用更精確的 DOM 掃描策略來還原對話全貌。

## 1. 記憶基礎：時間戳記 (Timestamps)
- **擷取要求**：`line_utils.extract_messages` 不得僅擷取文本，必須同步抓取對話氣泡旁的 `metaInfo-module__time` 或其對應 DOM。
- **作用**：時間戳記是模型判斷「任務是否已過時」或「是否為當下回覆」的核心依據。

## 2. 空間順序：倒序反轉 (Chronological Reversal)
- **特性**：LINE Chrome Extension 的 DOM 結構為「最新訊息在最上方」。
- **協議**：擷取組件必須在 JS 內部執行 `.reverse()`，確保 Python 接收到的列表順序為「舊到新」。嚴禁讓模型解析倒序歷史，這會導致 Deduplication 判定失敗。

## 3. 物理錨點：Prefix 辨識
- **身分識別**：依然強制使用 `[Hermes]` 作為物理前綴。
- **狀態重建**：`HistoryManager` 應掃描歷史中最近一個帶有 `[Hermes]` 前綴且內容與目前 `task` 高度匹配的訊息。
- **重複發訊防護**：若最新一則訊息是己方發出的 `[Hermes]` 訊息，系統必須強制進入「靜默守候」，嚴禁因自動重啟而再次發起任務。

## 4. 任務定義準則
- **單一目標化**：為了解決「資訊傾倒」問題，下達給代理人的 `task` 描述應拆解為單一子目標。
- **範例**：
    - 錯誤：`問訂位、給姓名、問停車、問電話`
    - 正確：`目前已確認時間，請僅詢問是否有停車位。`
