# LINE 訂位模擬任務失敗分析與準則

在 2026-05-10 的 Session 中，代理人在執行「娜比」訂位模擬時犯了多項嚴重錯誤，以下總結為應對準則。

### 1. 視窗選取失敗 (Window Context Error)
- **錯誤現象**：代理人長時間在包含「娜比」的群組中對話，而非點開個人私訊。
- **對策**：搜尋結果中若同時出現群組與個人，必須檢查標題是否帶有 `(n)` 人數標記。個人對話的標題僅會有名字。
- **操作**：使用狀態訊息（如「這個世界對我真的很好...」）作為唯一的點擊錨點，以確保進入正確的私訊視窗。

### 2. 腳本偏離 (Script Deviation)
- **錯誤現象**：放棄已測試的 `line_proxy.run_task (MCP)`，改用臨時寫的 `scripts/line_booking_agent.py`，導致邏輯無法應對複雜對話。
- **對策**：禁止自創腳本。`line-proxy/src/line_proxy.run_task (MCP)` 是唯一的真相來源。如果報錯，應 Patch `line_proxy.run_task (MCP)` 而不是新建文件。

### 3. 讀取過時訊息 (Stale Message Reading)
- **錯誤現象**：回報「娜比沒回覆」，但事實上對話框最下方已有新回覆。
- **原因**：LINE 的虛擬列表導致 `extract_messages` 在沒有捲動的情況下只抓到舊數據。
- **對策**：
    - 執行 `extract_messages` 前先進行 `await page.mouse.wheel(0, 5000)`。
    - 必須對比最後一則訊息的時間戳記與當前系統時間。
    - 使用 `vision_analyze` 進行人工雙重確認，不要只相信抓取到的文字陣列。

### 4. 模擬邏輯違和 (Simulation Inconsistency)
- **錯誤現象**：在店員（娜比）告知「沒車位」時，代理人竟回覆「感謝預訂」。
- **對策**：在模擬測試中，必須檢查「條件句」。若店員提出疑問或否定條件，代理人必須先解決該條件（詢問停車資訊或確認替代方案），嚴禁直接跳到結束語。
